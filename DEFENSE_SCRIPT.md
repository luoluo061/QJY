# 电商用户流失预测项目答辩讲稿

这份文档用于老师直接看项目、看代码、问实现思路时的讲解。表达重点不是“我跑出了结果”，而是“我自己设计并实现了一套完整的用户流失预测实验系统”。

---

## 1. 项目背景与研究问题

### 1.1 项目要解决什么问题

这个项目要解决的是电商场景中的用户流失预测问题。

具体来说，就是根据用户在过去一段时间内的行为数据，提前预测该用户在后续一段时间内是否还会继续活跃。如果能提前识别高风险用户，就可以用于后续的精准召回、优惠券投放、运营干预和用户分层管理。

### 1.2 这里的“流失”是什么意思

在这个项目里，流失不是指用户注销账号，而是指：

- 用户在预测期内完全没有产生任何行为

因此，这个项目本质上是一个二分类任务：

- `0`：未流失
- `1`：流失

### 1.3 这个项目的核心思路

项目的核心不是直接套一个分类模型，而是先把原始行为日志定义成一个严格的预测任务，再完成：

1. 数据审计
2. 时间窗口设计
3. 标签构建
4. 用户级特征工程
5. baseline 模型比较
6. 优化实验
7. 结果解释

所以这个项目真正的重点是“任务构造 + 特征设计 + 实验验证”，而不是单纯调用现成算法。

## 2. 数据来源与数据结构

### 2.1 原始数据是什么

原始数据位于：

- `archive/.csv`

这是电商用户行为日志数据，字段包括：

- `user_id`：用户编号
- `item_id`：商品编号
- `behavior_type`：行为类型编码
- `user_geohash`：用户地理位置编码
- `item_category`：商品类目编号
- `time`：行为发生时间，精确到小时

### 2.2 这类数据为什么不能直接建模

因为原始日志是“行为级数据”，不是“用户级数据”。

也就是说：

- 一条记录表示一次行为
- 一个用户会对应很多条记录

而分类模型要求样本粒度稳定，通常是“一行一个样本”。所以必须先把行为日志聚合成“每个用户一行”的宽表，再做监督学习。

### 2.3 这一步在代码里怎么实现

主逻辑在：

- `src/churn_pipeline/pipeline.py`

其中：

- `audit_dataset()` 负责审计原始数据
- `build_modeling_dataset()` 负责把行为流水转成用户级建模数据

## 3. 预测任务是怎么构造的

### 3.1 为什么要切时间窗口

用户流失预测和普通分类最大的区别是，它必须遵守时间顺序。

因为如果未来的数据提前进入了特征，模型效果会虚高，这种问题叫数据泄漏。

所以这个项目显式划分了：

- 观察期：用于提取特征
- 预测期：用于定义标签

### 3.2 当前项目的时间窗口

当前实验设置为：

- 观察期：`2014-11-18 00:00:00` 到 `2014-12-11 23:00:00`
- 预测期：`2014-12-12 00:00:00` 到 `2014-12-18 23:00:00`

也就是：

- 前 24 天用于构造用户画像
- 后 7 天用于判断用户是否流失

### 3.3 标签定义公式

对每个在观察期内出现过的用户 \(u\)，标签定义为：

\[
y_u =
\begin{cases}
1, & \text{如果用户 } u \text{ 在预测期内没有任何行为} \\
0, & \text{如果用户 } u \text{ 在预测期内仍然有行为}
\end{cases}
\]

这个定义的优点是：

- 业务含义清晰
- 能和时间窗严格对应
- 适合二分类模型处理

### 3.4 这一步在代码里怎么实现

对应函数：

- `choose_windows()`
- `build_modeling_dataset()`

项目中先确定预测期用户集合，再对观察期用户判断其是否在预测期出现，从而生成 `label_churn`。

## 4. 特征工程设计

### 4.1 特征工程为什么是这个项目的核心

原始行为日志不能直接喂给模型，所以这个项目最关键的一步就是把行为流水转换成用户级画像。

我把特征设计成四类：

- 活跃度特征
- 近期行为特征
- 行为结构特征
- 兴趣广度特征

这样做的逻辑是：

- 总量描述整体活跃程度
- 近期特征描述用户是否开始降温
- 结构特征描述行为习惯是否发生变化
- 广度特征描述兴趣面是否收缩

### 4.2 活跃度特征

1. `total_actions`

表示观察期内总行为次数。

\[
\text{total\_actions}_u = \sum_{t \in \text{观察期}} 1
\]

2. `active_days`

表示用户在观察期内发生过行为的去重天数。

\[
\text{active\_days}_u = |\{\text{date}(t)\}|
\]

3. `active_hours`

表示用户在观察期内活跃过的去重小时数。

\[
\text{active\_hours}_u = |\{\text{hour}(t)\}|
\]

4. `active_span_days`

表示用户首次行为到最后一次行为之间跨越的天数。

\[
\text{active\_span\_days}_u = \text{last\_day}_u - \text{first\_day}_u + 1
\]

这些变量反映的是用户是否持续活跃，而不是只看偶发行为。

### 4.3 近期行为特征

1. `last_7d_actions`

表示观察期最后 7 天的行为数：

\[
\text{last\_7d\_actions}_u = \sum_{t \in \text{观察期最后7天}} 1
\]

2. `last_3d_actions`

表示观察期最后 3 天的行为数：

\[
\text{last\_3d\_actions}_u = \sum_{t \in \text{观察期最后3天}} 1
\]

3. `recency_days`

表示用户最近一次行为距离观察期结束的间隔天数：

\[
\text{recency\_days}_u = \text{observation\_end} - \text{last\_timestamp}_u
\]

这类特征通常是流失预测中最关键的一组，因为流失最先表现为“最近不再活跃”。

### 4.4 行为结构特征

项目中对每种行为类型都统计了：

- 次数：`behavior_k_count`
- 占比：`behavior_k_ratio`

公式为：

\[
\text{behavior\_k\_count}_u = \sum I(\text{behavior\_type}=k)
\]

\[
\text{behavior\_k\_ratio}_u = \frac{\text{behavior\_k\_count}_u}{\text{total\_actions}_u}
\]

这类特征的意义是：

- 不只是看用户有没有活跃
- 还看活跃方式有没有变化

比如，如果用户过去主要是浏览，现在连浏览都没有，或者只剩少量单一行为，通常意味着更高的流失风险。

### 4.5 兴趣广度特征

1. `unique_items`

\[
\text{unique\_items}_u = |\{\text{item\_id}\}|
\]

2. `unique_categories`

\[
\text{unique\_categories}_u = |\{\text{item\_category}\}|
\]

3. `category_item_ratio`

\[
\text{category\_item\_ratio}_u = \frac{\text{unique\_categories}_u}{\text{unique\_items}_u}
\]

这类特征反映用户兴趣是否广泛、是否收缩，有助于识别“活跃还在，但兴趣面变窄”的用户。

### 4.6 比率类特征

为了减少不同用户行为总量差异带来的影响，项目还构造了多种比例特征：

1. `active_day_ratio`

\[
\text{active\_day\_ratio}_u = \frac{\text{active\_days}_u}{\text{observation\_days}}
\]

2. `last_7d_ratio`

\[
\text{last\_7d\_ratio}_u = \frac{\text{last\_7d\_actions}_u}{\text{total\_actions}_u}
\]

3. `last_3d_ratio`

\[
\text{last\_3d\_ratio}_u = \frac{\text{last\_3d\_actions}_u}{\text{total\_actions}_u}
\]

这些比例特征比绝对值更适合描述行为模式的变化趋势。

## 5. 算法选择与建模思路

### 5.1 为什么要比较多种模型

这个项目不是盲目堆模型，而是想回答一个更核心的问题：

- 在用户流失预测任务里，线性模型、Bagging 模型和 Boosting 模型各自表现如何

所以我选了四类具有代表性的算法：

- 逻辑回归 LR
- 随机森林 RF
- XGBoost
- LightGBM

### 5.2 逻辑回归 LR

逻辑回归是最标准的二分类基线模型，其概率形式为：

\[
P(y=1|x)=\sigma(w^Tx+b)
\]

其中：

\[
\sigma(z)=\frac{1}{1+e^{-z}}
\]

它的优点是：

- 可解释性强
- 适合观察特征是否存在明显线性区分信号

所以它在本项目里的角色主要是 baseline。

### 5.3 随机森林 RF

随机森林通过多棵决策树集成进行预测，其思想可以表示为：

\[
\hat{y} = \text{majority vote}(T_1(x), T_2(x), ..., T_n(x))
\]

它的优点是：

- 能建模非线性关系
- 对噪声和特征缩放相对不敏感
- 对结构化表格数据表现稳定

### 5.4 XGBoost

XGBoost 是提升树模型，其核心思想是逐轮学习残差：

\[
\hat{y}^{(t)} = \hat{y}^{(t-1)} + f_t(x)
\]

也就是每一轮新树都在纠正前一轮的错误。

它的优点是：

- 对复杂特征交互建模能力强
- 对表格数据任务通常表现优秀
- 可以结合不平衡样本权重做优化

### 5.5 LightGBM

LightGBM 也是提升树模型，和 XGBoost 类似，但在实现策略上更高效，适合较大规模的结构化数据。

在本项目中引入它，主要是为了与 XGBoost 形成对照，观察不同 boosting 实现的表现差异。

## 6. baseline 实验与优化实验

### 6.1 baseline 实验的作用

baseline 实验对应函数：

- `train_baseline_models()`

它的作用不是追求最好成绩，而是建立统一的比较基线。

也就是说，先在相同的数据划分和原始特征条件下比较各模型，看谁本身就更适合这个任务。

### 6.2 为什么还要做优化实验

如果只有 baseline，结论只能停留在“哪个模型初始表现更好”。

但实际建模中，还需要考虑：

- 噪声特征会不会影响模型
- 不平衡样本会不会让模型偏向多数类
- 默认参数是不是最优

所以项目单独设计了优化阶段。

### 6.3 特征筛选

项目使用互信息进行特征筛选：

\[
I(X;Y)
\]

互信息越高，表示特征与标签之间的信息关联越强。

这里的作用是：

- 去掉噪声特征
- 控制搜索复杂度
- 提高后续调参效率

### 6.4 不平衡样本处理

由于流失样本占比较低，所以项目中对不同模型做了类别不平衡修正：

- LR：`class_weight="balanced"`
- RF：`balanced_subsample`
- XGBoost：`scale_pos_weight = neg / pos`
- LightGBM：`class_weight="balanced"`

其中 XGBoost 的权重公式为：

\[
\text{scale\_pos\_weight} = \frac{\text{负样本数}}{\text{正样本数}}
\]

作用是：

- 提高模型对少数类流失用户的关注度
- 避免模型只偏向未流失用户

### 6.5 参数优化

项目使用：

- `RandomizedSearchCV`

统一以 `ROC_AUC` 作为搜索目标，在给定搜索空间中寻找更优参数组合。

这样做的意义是：

- 保证不同模型在统一标准下比较
- 减少人工试参数的不确定性

## 7. 模型评估指标怎么解释

### 7.1 为什么不能只看准确率

因为这是一个明显的不平衡分类任务，流失用户占比较低。

如果模型把大多数用户都预测为未流失，准确率可能仍然不低，但这对流失识别没有价值。

### 7.2 本项目使用的主要指标

1. `Precision`

\[
\text{Precision} = \frac{TP}{TP+FP}
\]

表示预测为流失的用户中，真正流失的比例。

2. `Recall`

\[
\text{Recall} = \frac{TP}{TP+FN}
\]

表示真正流失的用户中，被识别出来的比例。

3. `F1`

\[
F1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}
\]

表示精确率和召回率的平衡。

4. `ROC_AUC`

表示模型整体区分正负样本的能力，越大越好。

### 7.3 这些指标在项目里的意义

- `ROC_AUC`：看整体区分能力
- `Recall`：看能抓住多少真正流失用户
- `Precision`：看误报率高不高
- `F1`：看在不平衡场景下整体平衡效果

## 8. 当前实验结果怎么讲

### 8.1 baseline 结果

当前 baseline 阶段按 `ROC_AUC` 最优的是：

- `LR`

这说明：

- 当前特征体系下已经存在较明显的线性可分信号
- 即使不做复杂优化，用户行为特征本身已经能区分部分流失用户

### 8.2 优化后结果

优化阶段按 `ROC_AUC` 最优的是：

- `XGBoost`

优化阶段按 `F1` 最优的是：

- `RF`

这说明：

- 如果强调整体区分能力，XGBoost 更强
- 如果强调少数类识别的平衡性，RF 的 F1 更有优势

### 8.3 这组结果反映了什么

它说明这个任务里同时存在两类规律：

- 一部分是线性可分信号，所以 LR baseline 不差
- 一部分是复杂非线性关系，所以优化后树模型优势更明显

## 13. 老师最可能追问的问题与回答

### 问题 1：为什么不能直接用原始日志训练

回答：

因为原始日志是一条行为一行，而分类模型要求样本粒度稳定。这里必须先把日志统一聚合成“每个用户一行”的建模宽表，否则样本定义本身就是混乱的。

### 问题 2：为什么一定要切观察期和预测期

回答：

因为流失预测本质上是时间相关任务。特征必须来自过去，标签必须来自未来。如果未来行为提前进入特征，就会造成数据泄漏，结果不真实。

### 问题 3：为什么把预测期无行为定义成流失

回答：

因为这样定义最容易与时间窗直接对应，也最适合转成可计算的二分类标签。它既有业务含义，也有实现上的明确边界。

### 问题 4：为什么要比较四种模型

回答：

因为逻辑回归、随机森林和两类提升树代表了不同的归纳方式。我要验证这个任务里到底是线性规律更重要，还是复杂非线性关系更重要。

### 问题 5：为什么不能只看准确率

回答：

因为流失用户占比低，准确率会被多数类掩盖。即使模型几乎都预测成未流失，准确率也可能不差，但识别流失用户的能力会很弱。所以必须看 `Recall`、`F1` 和 PR 曲线。

### 问题 6：为什么优化后树模型更好

回答：

因为树模型更擅长利用非线性关系和特征交互。在经过特征筛选并处理不平衡之后，它们对复杂行为模式的利用更充分，所以表现提升更明显。

### 问题 7：为什么近期活跃相关特征最重要

回答：

因为用户是否流失，最直接的前兆就是近期活跃度下降。比如最近 7 天行为减少、最近一次行为距离现在更远，这些信号往往最直接反映用户流失风险。

## 14. 一段适合答辩时直接说的总结

这个项目的核心不是简单地调用几个分类模型，而是先把原始电商行为日志设计成一个严格按时间顺序定义的流失预测任务，再通过用户级特征工程把离散行为转成可学习的用户画像，最后比较不同模型并进行优化实验。从结果上看，原始特征已经具备较强区分能力，而经过特征筛选和不平衡处理后，树模型进一步体现出了优势。整体上，这个项目完成的是一套从数据到结果再到解释的完整实验系统。
